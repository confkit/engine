# Basic Standards

实施和重构时, 默认追求干净结构、明确边界和可持续维护.

## 通用编码风格

- **生产级实现** - 默认交付生产级代码, 不写过渡性脚手架
- **明确表达** - 优先用结构、命名和清晰逻辑表达语义, 不依赖暧昧注释补救
- **可读优先** - 可读、可维护、可验证优于炫技式简写
- **一致优先** - 优先遵循项目现有模式, 不为局部新意破坏整体一致性
- **职责清楚** - 一个模块只解决一类问题, 避免边界漂移和职责交叉

## 面向开发的模块设计

- **面向开发** - 模块、功能、抽象、实现以便于开发、维护、排查问题为第一原则
- **抽象有因** - 只为真实复用或边界隔离做抽象, 不做空洞设计
- **边界失真时重整** - 职责交叉、依赖混乱、边界模糊时主动重整结构
- **避免中性容器** - 不把无关逻辑堆进 `utils`、`common`、`misc` 之类的模糊位置

## 配置与硬编码

- 先检查项目是否已有常量或配置管理机制
- 已有机制时接入现有位置, 不新开第二套
- 无现有机制时再创建统一管理位置
- 示例代码、演示片段、mock 数据可使用硬编码
- 临时硬编码必须标记 `// TODO: RISK - [原因描述]`

## 默认不做兼容层

除非开发者明确要求, 默认不主动添加:

- 向后兼容垫片
- 迁移桥接层
- 为旧实现保留的过渡分支
- "旧实现"、"遗留代码" 之类的过渡注释

目标: 生产级代码, 而非过渡性脚手架.

## 跨文件一致性

任何变更确认后, 同步检查:

- 相关文件是否需要一起调整
- 测试是否需要同步更新
- 类型定义是否需要同步调整
- 相关设计文档是否需要同步更新

## 结构与模块

- 目录、文件、模块层级必须直观表达职责
- 优先按职责、边界、协作关系组织代码
- 文件过大、职责混杂、入口不清时继续拆分
- 抽象必须服务于真实复用或边界隔离
- 职责交叉、依赖混乱时优先重整结构

## 命名

- 命名必须清楚表达职责和用途
- 优先业务语义、输入输出语义或行为结果语义
- 避免 `data`、`info`、`item`、`manager`、`handler`、`util`、`common` 等弱语义名称, 除非上下文已经足够明确
- 如果名称始终不准确, 优先调整结构或重新建模

## 文本表达

- 状态、提示、语义优先使用明确文本、命名或结构表示
- 不用 emoji 承载代码语义或状态表达

## TODO 标记

对不确定或不完整的内容, 统一使用:

```ts
// TODO: [描述] - Context: [为什么不确定]
// TODO: RISK - [风险描述]
```

使用场景:

- 需求模糊
- 边界情况需验证
- 集成点不明确
- 临时硬编码
