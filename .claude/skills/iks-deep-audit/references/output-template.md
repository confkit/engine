# Deep Audit Output Template

## 输出顺序

1. 审查范围
2. `Critical`
3. `High`
4. `Medium`
5. 整体结论
6. 优先处理顺序

## 审查范围

先明确:

- 语言
- 检查模式: 全局 / `--scope` / `--diff` / `--scope + --diff`
- focus: 无 / `crash` / `failure` / `state` / `flow` / `resource` / `structure`
- 实际覆盖的目录、模块、文件或变更范围

## Findings 模板

每个 finding 使用统一结构:

```md
### [Critical|High|Medium] <简短标题>

- 位置: <文件 / 模块 / 链路>
- 问题: <风险点本身>
- 原因: <为什么会出问题>
- 触发条件: <什么情况下暴露>
- 影响范围: <用户面 / 系统面 / 维护面影响>
- 修复方向: <优先给局部、可落地的修复思路>
```

## 整体结论

- 用 2 - 4 条总结项目或范围内最关键的健康结论
- 明确当前更偏崩溃风险、结构风险、状态风险还是性能风险
- `--diff` 模式下区分“本次引入”与“原本存在”
- `--focus` 模式下说明该 focus 是否确实是当前主风险

## 优先处理顺序

- 先处理 `Critical`
- 再处理会扩大故障面的 `High`
- 最后处理 `Medium` 中会持续累积的结构和资源问题
- 不要把低收益优化排到高风险问题前面
