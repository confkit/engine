# Go Deep Audit

## 重点关注

- panic、请求失控、阻塞链路和恢复路径缺失
- goroutine 生命周期、channel 关闭职责和 context 传播
- handler 过重、边界失真、无界并发和锁粒度过粗
- 高分配、重复序列化、日志污染和延迟敏感路径退化

## 核心检查项

### 崩溃与异常风险

- 是否存在 panic 风险、未处理错误和恢复路径缺失
- 超时、取消、重试、降级是否清晰, 是否会演变成请求堆积
- goroutine 是否有明确生命周期和退出路径
- channel 阻塞、关闭职责和资源释放是否可靠

### 并发与请求链路

- `context.Context` 是否沿链路正确传递
- 是否存在过度 goroutine 化、过度 fan-out / fan-in 或无界并发
- handler 是否同时承担协议解析、业务处理、数据转换和 IO
- 是否存在阻塞式数据库、缓存、RPC 串行链路

### 结构与边界

- 是否把状态和职责堆进大 struct、大 service 或大 handler
- 是否存在不必要接口层、间接调用链和空洞抽象
- 热路径、控制流、冷路径是否被区分对待
- 目录结构、命名、模块职责是否清晰表达边界

### 性能与资源风险

- 热路径是否频繁分配切片、map、字符串和临时对象
- 已知容量是否缺少 `make(..., capacity)`
- 是否存在无意义字符串拼接、字节切片复制、接口装箱
- 锁作用域是否过大, 是否把业务逻辑包进锁内

### IO、日志与数据处理

- 热路径是否存在同步日志、频繁格式化、重复 JSON 编解码
- 是否缺少批处理、缓存、结果复用和请求去重
- 小数据场景是否机械使用 map 而不是 slice
- 是否为调试长期保留高成本输出或 tracing

## 工具建议

- `pprof`
- `go tool trace`
- benchmark / `benchstat`
