# Deep Audit Core

## 审查定位

- 不是只看最近变更的 review
- 不是单纯性能检查
- 是对项目、模块或变更范围做系统性风险审查
- 优先判断是否存在会导致崩溃、严重异常、状态失控和长期失衡的问题

## 共用风险模型

可用于 `--focus` 的风险类别:

- `crash` - 崩溃、panic、ANR、白屏、越界、非法状态
- `failure` - 不可恢复异常、错误吞没、恢复路径缺失、清理失败
- `state` - 状态重复源、状态失控、生命周期错配、恢复不一致
- `flow` - 主链路阻塞、事件循环失控、并发模型失衡、异步竞态
- `resource` - 热路径分配、资源浪费、日志污染、重复计算、无意义刷新
- `structure` - 职责边界不稳、抽象过重、模块失衡、可维护性恶化

### Critical

- 崩溃、panic、ANR、白屏、越界、非法状态
- 不可恢复异常、terminal 状态损坏、请求链路失控
- 明确的数据损坏、状态破坏、资源泄漏或清理失败

### High

- 主链路阻塞、事件循环失控、并发模型失衡
- 生命周期错配、状态重复源、恢复路径缺失
- 明显不可接受但不一定立刻崩溃的结构和行为问题

### Medium

- 热路径低效、资源浪费、日志污染、重复计算
- 职责边界不稳、抽象过重、模块可维护性变差
- 暂时可运行, 但长期会恶化的问题

## 主动与被动的关系

- 编码期规则负责预防风险
- deep audit 负责放大风险、解释风险和按严重度排序
- 发现问题时, 要同时判断是否属于日常规则已覆盖但被违背

## 输出格式

每个 finding 至少包含:

- **位置** - 文件、模块、目录或链路
- **级别** - `Critical` / `High` / `Medium`
- **问题** - 风险点本身
- **原因** - 为什么会出问题
- **触发条件** - 什么情况下会暴露
- **影响范围** - 用户面、系统面或维护面影响
- **修复方向** - 优先给局部、可落地的修复思路

如果使用 `--focus`:

- 在审查范围中明确当前 focus 类别
- 优先展开该类别的问题
- 但不得因为 focus 而漏掉其他类别的 `Critical`

## 审查边界

- 不为了“凑数量”而罗列低价值问题
- 不为了猜测性优化建议做大规模重构
- 先抓真正不可接受的问题, 再谈性能和结构优化
- 如果范围是 `--diff`, 要区分“本次引入”与“原本就存在”的问题
