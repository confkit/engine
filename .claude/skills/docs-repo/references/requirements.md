# Requirements Template

## 适用场景

用于功能需求、业务改造、流程调整这类“先定目标与边界, 再进入设计”的文档.

## 文件建议

- 目录下固定保留一个简洁的 `overview.md`, 只写范围、导航和阅读顺序
- 文件名使用英文 slug, 例如 `login.md`, `checkout-refund.md`
- 一个文件聚焦一个主题, 不在同一份文档里混写多个无关需求

## 推荐结构

```md
# <feature-title>

## 背景

## 目标
- ...

## 用户流程 / 页面结构
### <flow-or-page>
- ...

## 功能要求
| 编号 | 说明 | 验收标准 |
| --- | --- | --- |
| FR-1 | ... | ... |

## 边界与约束
- ...

## 非目标
- ...
```

## 编写约束

- requirement 关注目标、流程、边界、验收, 不展开技术实现
- 功能要求优先按业务域、页面或用户流程分组
- 验收标准必须可验证, 避免“体验更好”这类空泛描述
- 需要补充上下文时, 写在“背景”或“边界与约束”, 不塞进功能要求表
- 若某个流程较复杂, 单独拆 `###` 小节说明, 不压成一行字段
