# Architectures Template

## 适用场景

用于技术方案、模块拆分、目录设计、状态与接口边界这类“为什么这样设计”的文档.

## 文件建议

- 目录下固定保留一个简洁的 `overview.md`, 只写范围、导航和阅读顺序
- 文件名使用英文 slug, 例如 `auth-state.md`, `monorepo-packages.md`
- 一个文件聚焦一个方案主题, 不把多个独立设计混写在一起

## 推荐结构

````md
# <architecture-title>

## 背景与目标

## 方案概览

## 模块职责
| 模块 | 职责 | 不负责 |
| --- | --- | --- |
| ... | ... | ... |

## 关键结构
~~~text
<directory / module / request-flow>
~~~

## 关键流程
1. ...
2. ...

## 设计原则与约束
- ...

## 取舍与风险
| 主题 | 结论 | 影响 |
| --- | --- | --- |
| ... | ... | ... |
````

## 编写约束

- architecture 必须显式回答“为什么这么拆, 为什么不那样拆”
- 模块边界、目录边界、接口边界优先写清楚, 再谈实现细节
- 优先用表格、结构图、流程步骤表达关系, 不写成字段卡片
- 若涉及 monorepo、分包、目录拆分, 必须能直观看出职责归属
- 取舍要写清收益、代价和约束, 不只写单向结论
