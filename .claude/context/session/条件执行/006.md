# 议题 006: 条件执行日志和可见性 - 会话上下文

## 议题概述
实现条件执行的日志记录和用户可见性，提供清晰的条件评估过程和统计信息。

## 深度分析结果
- **日志架构评估**: 现有EventHub + TaskLogger系统完全适合条件日志需求
- **集成策略**: 无缝集成现有日志基础设施，最小化代码变更
- **性能影响**: 微乎其微，主要是微秒级计时和可控的I/O开销

## 核心技术要求
1. **分层日志设计**: TaskLogger(用户级) + tracing(调试级)的双层日志架构
2. **统计信息管理**: 扩展StepResult和Task实现条件执行统计
3. **用户体验优化**: 设计清晰直观的日志输出格式
4. **调试信息完善**: 在ConditionEvaluator中添加详细的评估过程日志
5. **错误信息优化**: 提升条件解析错误的用户友好性

## 技术决策记录
- **日志层级**: TaskLogger(info/error) 用于用户级信息，tracing(debug/trace) 用于开发调试
- **统计方式**: 扩展现有Task统计机制，复用get_execution_stats()模式
- **性能优化**: 使用Instant进行高精度计时，trace级别仅调试模式启用
- **格式设计**: 任务日志简洁明了，控制台日志详细调试

## 核心实现方案

### 日志层级设计
```rust
// TaskLogger (写入文件) - 用户关心的核心信息
TaskLogger::info()  -> 条件表达式、评估结果、跳过原因
TaskLogger::error() -> 条件解析错误、类型转换错误

// tracing (控制台) - 开发调试信息  
tracing::debug!()   -> 条件评估耗时、变量解析结果
tracing::trace!()   -> 详细的求值步骤、递归层级
```

### 统计信息扩展
```rust
pub struct ConditionStats {
    pub total_conditions: usize,
    pub executed_by_condition: usize,
    pub skipped_by_condition: usize,
    pub condition_errors: usize,
    pub avg_evaluation_time_us: f64,
}
```

### 日志输出格式
```
任务日志格式（用户友好）：
[1/5] ○ Step 'deploy' skipped (condition: ${ENV} == 'test' -> false)

控制台调试格式（开发友好）：
DEBUG condition: Evaluating '${ENV} == "test"' -> false (12μs)
TRACE condition: └─ Variable resolution: ENV = "prod"
```
