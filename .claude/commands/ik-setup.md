# 项目搭建

在整体规划已经定案后, 仅搭建项目骨架, 不进入具体功能实现.

## 调用方式

```bash
/ik-setup
/ik-setup --feature <name>
/ik-setup --module <name>
```

## 适用场景

- 需求文档和技术设计已基本定案
- 当前目标是先搭好接口、状态、拆包、目录、模块边界
- 需要为后续长期维护和扩展预留稳定结构
- 暂时不实现完整业务流程, 只落骨架和占位

## 不做的事

- 不补做需求讨论或技术选型
- 不重写已定案的规划文档
- 不直接实现完整功能逻辑
- 不为了“看起来能跑”而写大量临时代码

## 前置要求

- 必须先读取并遵循 `basic-mindset` skill
- 必须先确认需求、架构、todo 已基本定案
- 如果规划仍存在关键分歧, 先回到 `ik-plan`, 不继续搭建

## 工作目标

- 搭好包结构、目录结构、模块边界
- 搭好接口、状态模型、领域抽象、基础 wiring
- 搭好后续功能可持续扩展的实现骨架
- 保持骨架直观、可导航、可继续分阶段填充

## 工作流程

1. **读取规划** - 先读 requirements、architecture、todo 等已定案文档
2. **识别骨架边界** - 明确哪些目录、包、模块、接口、状态需要先落位
3. **检查复杂点** - 如果发现某个功能域、子系统、跨模块协作过于复杂, 不能笼统并入整体骨架
4. **拆出子问题** - 对复杂功能域单独拆模块或子目录, 必要时补独立骨架占位
5. **调用 subagent** - 遇到复杂拆包、复杂状态边界、复杂模块协作时, 主动调用 subagent 给出备选方案并提醒开发者
6. **确定抽象层** - 先明确模块职责、依赖方向、扩展点, 再开始写骨架代码
7. **搭建骨架** - 创建目录、包、接口、状态、模块入口、占位实现
8. **校验长期合理性** - 检查结构是否直观、是否符合长期维护和扩展需求

## 复杂点处理

以下情况不要硬塞进一个整体性设计中, 需要单独拆开考虑:

- 某个功能域明显有独立状态流和生命周期
- 某个模块未来会独立扩展或替换实现
- 某段交互链路跨越多个技术层
- 某个子系统已经接近独立包或独立应用
- 某个能力同时涉及接口、状态、存储、任务调度等多重边界

识别到这类问题时:

- 及时指出复杂点
- 给出拆分理由
- 通过 subagent 补充方案建议
- 提醒开发者决定是否独立建模块、子包、子目录或单独设计

## 技术设计要求

技术设计阶段必须先做好抽象设计, 尤其重视模块间设计原则:

- **开闭原则** - 扩展新能力时, 优先新增模块、实现或适配层, 不靠频繁修改核心骨架
- **依赖方向清晰** - 高层依赖抽象, 低层提供实现, 不反向耦合
- **职责边界明确** - 目录、包、文件、模块直观表达职责
- **结构可导航** - 看到目录就能理解项目分层和主流程
- **长期可扩展** - 先为未来功能域、实现替换、测试隔离留出结构空间

## 结构拆分要求

根据项目形态选择合理的拆分粒度:

- monorepo: 先明确应用层、共享层、领域层、基础设施层的拆包边界
- 单应用: 先明确模块目录、领域目录、共享目录、入口目录的边界
- 共享代码: 只有在存在真实复用或明确边界时才上提
- 文件拆分: 一个文件只承载一类职责, 避免骨架阶段就堆成巨型入口

## 输出要求

- 输出应以项目骨架代码为主, 不改动已定案的规划文档
- 新增接口、状态、模块入口、抽象层时, 必须保持命名直观
- 占位实现必须明确表达职责, 不用含糊命名和临时拼接结构
- 若某部分暂不实现, 用清晰占位而不是伪完整逻辑

## 交付检查

- 目录结构是否直观表达项目分层
- 包与模块边界是否清晰
- 接口和状态是否能支撑后续扩展
- 是否严格遵循 `basic-mindset` 的模块间设计原则
- 是否把复杂子问题单独拆开, 而不是强行塞回主骨架
