# IK Plan

项目方案规划讨论, 收集项目需求并输出结构化设计文档.

## 调用方式

```bash
/ik-plan
/ik-plan --feature <name>
/ik-plan --fix <bug-name>
/ik-plan --update <name>
```

## 模式选择

根据命令参数选择工作模式:

| 模式           | 参数               | 说明                                       |
| -------------- | ------------------ | ------------------------------------------ |
| **项目初始化** | 无参数             | 新项目从零开始, 收集完整需求和技术选型     |
| **新增功能**   | `--feature <name>` | 为现有项目添加新功能, 先讨论再输出规划文档 |
| **问题修复**   | `--fix <bug-name>` | 为复杂问题修复制定分析和实施计划           |
| **更新文档**   | `--update <name>`  | 更新现有功能的设计和文档                   |

## 项目初始化模式

### 1. 需求了解

在进入技术选型前, 先充分了解项目需求:

- **业务背景** - 项目的业务场景和目标用户
- **核心问题** - 项目要解决的核心问题
- **功能范围** - 必须功能 vs 可选功能 vs 未来考虑
- **非功能需求** - 性能、安全、可用性等要求
- **约束条件** - 时间、预算、技术栈限制等

### 2. 项目信息收集

分两阶段对话:

**第一阶段 - 基础信息采集**

- **项目规模**: 个人 / 小团队 / 企业级
- **维护长久度**: 短期项目 / 长期维护 / 持续演进
- **项目类型**: Web / 移动 / 桌面应用 | 命令行工具 | API 服务 | 工具库 | 其他

**第二阶段 - 技术选型**

根据需求和信息, 提供当下流行的技术方案供开发者选择, 涵盖:

- **开发语言**: 主流语言选项及适用场景
- **UI 技术**: 前端框架、UI 库、主题方案
- **数据存储**: 数据库选型、缓存方案
- **部署运维**: 部署方式、容器化、CI / CD

### 3. 方案讨论

与开发者进行方案讨论, 确认技术选型和实现路径.

若方案涉及全局配置、环境变量、运行参数或设置中心, 在讨论阶段先按需读取 `basic-config-system` 的 `references/layering.md`, 提出方案, 与开发者确定方案

### 4. 文档输出

定案后根据情况决定需要输出哪些文档类型, 再按文档类型决定走脚本还是走模板. `ik-plan` 负责规划内容和产出决策, 不负责手工维护文档路径、命名、ID 或桶文件规则.

需要输出的文档类型主要包括:

**`.docs/requirements`** - 需求设计:

- 承载目标、主流程、关键状态、边界 / 异常和验收标准

**`.docs/architectures`** - 技术设计:

- 承载技术方案、模块职责、关键结构、约束和取舍结论

**`.docs/todo`** - 任务文档:

- 承载实施拆分、责任域、依赖关系和阶段性验证

**`.docs/testing`** - 完整测试用例:

- 承载测试环境、测试场景矩阵、关键步骤和预期结果

**`.docs/issues`** - 问题与优化记录:

- 承载 bug / 优化点记录、影响范围和处理方向
- issue 只负责记录和跟踪, 若进入处理必须先生成关联 todo

### 文档动作决策

`ik-plan` 不只决定文档类型, 还要决定每类文档走哪条产出路径:

| 场景 | 模板型文档 | 流程型文档 |
| --- | --- |
| 项目初始化 | 新建 `requirement` / `architecture` / `testing` 文档 | `create todo`, 必要时 `validate` |
| 新增功能 | 先看现有文档, 再决定新增或更新 `requirement` / `architecture` / `testing` | 视实施范围决定是否 `create todo`, 必要时 `validate` |
| 问题修复 | 默认不更新 `requirement` / `architecture` / `testing` | 先 `create issue`, 再 `issue plan`, 必要时 `validate` |
| 更新文档 | 定位现有文档后直接更新对应模板型文档 | 若牵涉实施跟踪, 再决定是否同步调整 `todo / issue` |

动作约束:

- `ik-plan` 必须先决定文档类型, 再决定该类型是“模板直写”还是“脚本操作”
- `requirement / architecture / testing` 不走 `docsctl` CRUD, 而是读取对应 reference 后直接编写或更新文档
- fix 模式默认是 `create issue -> issue plan`, 不是直接 `create todo`
- 只有 `todo / issue` 需要通过 `docsctl validate` 做 schema 校验
- 需要查 `todo / issue` 的现状、详情、关联关系时, 直接调用 `docs-repo` 的 `docsctl`
- 需要调整模板型文档时, 先读取现有文档和对应 reference, 不把模板型文档误当成脚本实体
- `todo / issue` 的创建、更新、关闭、删除统一走 `docs-repo` 的脚本能力
- 需要一次生成完整 todo 拆解时, 优先使用 `todo batch-create`、`todo batch-append` 或 `issue batch-plan`
- 模板型文档只需遵循 `docs-repo` reference 与当前文档规范, 不强行套脚本字段

## 新增功能模式

新增功能模式下, 需求讨论必须尽可能细, 先把目标讲透, 再进入技术设计, 避免实现阶段偏离目标.

1. **明确功能目标** - 先确认业务价值、目标用户、核心收益和非目标
2. **细化使用场景** - 拆主要用户流程、关键操作路径、异常路径和边界场景
3. **明确交互与结果** - 确认页面 / 模块入口、关键状态、用户可见反馈、成功 / 失败结果
4. **明确数据与规则** - 确认输入输出、字段变化、业务规则、权限边界、约束条件
5. **明确验收标准** - 把“做完算完成”讲清楚, 明确必须满足的验收条件和不可接受结果
6. **分析影响范围** - 梳理涉及的模块、接口、数据结构、状态流、文档和测试范围
7. **讨论方案取舍** - 与开发者讨论实现路径、边界、风险、复杂点和是否需要拆子设计
8. **定案后输出文档** - 根据讨论结果更新或新增 requirements、architecture、todo、testing 文档

### 讨论约束

- 不要只停留在“要做什么”, 必须继续追问“怎么用”、“何时算成功”、“哪些情况算失败”
- 对目标、边界、状态、权限、异常、验收不清楚时, 不要过早进入技术设计
- 发现开发者表述仍偏抽象时, 主动把需求拆成更具体的流程、场景和规则
- 若某个功能域明显复杂, 需要单独拆出子问题继续讨论, 不要笼统并入一个总方案
- 若功能涉及配置系统, 在方案讨论时主动确认配置分层、密钥边界、覆盖优先级和长期维护方式
- 在需求没有收敛前, 不输出看似完整但实际建立在猜测上的设计文档

### 输出要求

- requirements 文档必须足够支撑后续实现, 让实现阶段不靠临场猜测补需求
- requirements 文档至少覆盖目标、主流程、关键状态、异常 / 边界、验收标准
- requirements 文档可根据变更范围选择新增或更新
- architecture 文档必须明确方案概览、模块职责、关键结构、约束和取舍
- todo 文档必须能支撑实施拆解、责任域跟踪和阶段性验证
- testing 文档若需要输出, 必须按环境准备、场景矩阵、关键用例组织
- 若规划过程中同时沉淀问题 / 优化点, issue 只负责记录, 进入处理前再展开关联 todo
- `todo / issue` 的 schema、命名、ID、分桶、校验和 CRUD 统一交由 `docs-repo` 处理
- 模板型文档的结构、章节和编写方式统一遵循 `docs-repo` 对应 reference

## 问题修复模式

适用于复杂问题修复, 如根因不明、影响范围大、涉及多模块联动的问题.

1. **明确问题现象** - 收集报错、复现条件、用户影响和紧急程度
2. **分析根因范围** - 梳理可能根因、相关模块、调用链和边界条件
3. **讨论修复方案** - 与开发者讨论排查路径、修复策略、风险和回归范围
4. **输出修复计划** - 先沉淀 issue, 再生成关联 todo; 不更新 requirements、architecture、testing 等规划文档

### 输出要求

- fix 模式不要求更新需求或设计文档
- fix 模式先沉淀 issue, 再生成关联 todo, 不允许 issue 直接作为实施单元
- issue 必须包含现象 / 问题、处理方向、影响范围和关联 todo 字段
- issue 关闭或删除时, 后续执行阶段必须先检查关联 todo 状态; 忽略未完成 todo 时必须给出理由
- todo 文档中必须包含复现条件、修复步骤、回归检查点
- fix 相关文档的 CRUD 与状态变更统一通过 `docs-repo` 的脚本执行

## 更新文档模式

1. **定位现有文档** - 找到功能对应的设计和文档
2. **分析变更内容** - 理解需要更新的部分
3. **同步更新** - 保持文档间的一致性

## 文档编写要求

- 输出某一类文档前, 先按需读取 `docs-repo` 对应 reference, 不要一次性加载全部文档模板
- `todo / issue` 的 CRUD、命名、ID、分桶、校验统一通过 `docs-repo` skill 及其脚本处理
- `requirement / architecture / testing` 按模板直接编写, 文件名使用英文 slug, 一个文件只承载一个主题
- `requirement / architecture / testing` 目录固定保留一个简洁的 `overview.md`, 其余主题文档按英文 slug 命名
- `ik-plan` 只负责明确需要哪些文档、每类文档应承载什么内容, 不主动维护流程型文档的底层规则
- 需要结构化结果供后续 agent 消费时, 优先让 `docs-repo` 的脚本输出 `--json`
- 设计文档: 背景 / 目标 + 方案说明 + 模块职责 + 关键表格 / 结构图 + 取舍结论
- 需求文档: 按业务域 / 页面 / 用户流程拆分, 先写结构, 再写功能要求 / 边界 / 约束
- Todo 文档: 承载实施拆解、责任域、依赖和验证要求
- 测试文档: 先写环境准备, 再写测试场景矩阵, 关键场景补执行步骤和预期结果
- Issues 文档: issue 负责记录和跟踪, 进入处理前必须先生成关联 todo

## 计划约束

- 非少量小改动时, 先给出实施计划再进入实现
- 计划必须可追踪、可验证, 并在任务推进过程中持续保持可见
- 文档定案后若只需先搭项目骨架, 使用独立命令 `ik-setup`, 不把落地混进 plan
